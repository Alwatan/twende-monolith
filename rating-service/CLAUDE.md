# CLAUDE.md — Rating Service

> Manages mutual ratings between riders and drivers after completed rides.
> Read this fully before writing any code in this module.

---

## 1. What This Service Does

The rating service handles post-ride ratings where riders rate drivers and drivers rate
riders. Each ride allows exactly one rating per role (one from the rider, one from the
driver). Rating aggregates are cached in Redis and exposed via an internal endpoint used
by matching-service for driver quality scoring.

**Port:** 8092
**Database:** `twende_ratings`

---

## 2. API Endpoints

| Method | Path | Description |
|---|---|---|
| `POST` | `/api/v1/ratings` | Submit a rating after a completed ride |
| `GET` | `/api/v1/ratings/driver/{driverId}` | Driver's public rating summary |
| `GET` | `/api/v1/ratings/me` | Calling user's own rating summary |
| `GET` | `/internal/ratings/driver/{driverId}/score` | Driver score (used by matching-service) |

### Submit Rating Request

```json
POST /api/v1/ratings
{
  "rideId": "uuid",
  "score": 5,
  "comment": "Very smooth ride!"
}
```

The service infers `ratedUserId` and `raterRole` from the ride record and the calling
user's identity (via `X-User-Id` and `X-User-Role` headers injected by the API Gateway).

- If `X-User-Role` is `RIDER`, the rated user is the driver from the ride record.
- If `X-User-Role` is `DRIVER`, the rated user is the rider from the ride record.

### Submit Rating Response

```json
{
  "success": true,
  "data": {
    "id": "uuid",
    "rideId": "uuid",
    "ratedUserId": "uuid",
    "raterUserId": "uuid",
    "raterRole": "RIDER",
    "score": 5,
    "comment": "Very smooth ride!",
    "createdAt": "2026-04-07T12:00:00Z"
  }
}
```

### Driver Rating Summary Response

```json
GET /api/v1/ratings/driver/{driverId}
{
  "success": true,
  "data": {
    "driverId": "uuid",
    "averageScore": 4.7,
    "totalRatings": 312,
    "distribution": {
      "5": 210,
      "4": 65,
      "3": 25,
      "2": 8,
      "1": 4
    }
  }
}
```

### Internal Driver Score Response

```json
GET /internal/ratings/driver/{driverId}/score
{
  "driverId": "uuid",
  "average": 4.7,
  "count": 312
}
```

This is a lightweight response used by matching-service for match scoring. No
`ApiResponse` wrapper on internal endpoints.

## 4. Database Schema

```sql
-- V1__create_ratings_schema.sql

CREATE TABLE ratings (
    id            UUID        PRIMARY KEY,
    country_code  CHAR(2)     NOT NULL,
    ride_id       UUID        NOT NULL,
    rated_user_id UUID        NOT NULL,
    rater_user_id UUID        NOT NULL,
    rater_role    VARCHAR(10) NOT NULL CHECK (rater_role IN ('RIDER', 'DRIVER')),
    score         SMALLINT    NOT NULL CHECK (score BETWEEN 1 AND 5),
    comment       TEXT,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE(ride_id, rater_role)
);

CREATE INDEX idx_ratings_rated_user ON ratings(rated_user_id);
CREATE INDEX idx_ratings_rater_user ON ratings(rater_user_id);
CREATE INDEX idx_ratings_ride ON ratings(ride_id);
```

**Key constraints:**
- `UNIQUE(ride_id, rater_role)` enforces one rating per ride per role.
- `score` is constrained to 1-5 at the database level.
- `id` is ULID-generated by the application via `UlidGenerator` (no DB default needed).

## 6. Rating Aggregation and Caching

Aggregates are computed from the database and cached in Redis as a JSON hash:

```
KEY:   rating:driver:{driverId}
VALUE: { "average": 4.7, "count": 312, "updatedAt": "2026-04-07T12:00:00Z" }
TTL:   1 hour
```

**On every new rating submission:**
1. Save the rating to the database.
2. Recompute the aggregate for the rated user from the database:
   ```java
   Double avg = ratingRepository.findAverageScoreByRatedUserId(ratedUserId);
   Long count = ratingRepository.countByRatedUserId(ratedUserId);
   ```
3. Update the Redis cache with the new aggregate.
4. Publish `twende.ratings.submitted` Kafka event.

**Cache miss on read:**
When the internal endpoint or public endpoint is called and the cache is empty,
compute the aggregate from the database and populate the cache.

**At scale:** Replace the per-submission recomputation with a PostgreSQL materialized
view refreshed on a schedule. For now, direct computation is sufficient.

## 7. Kafka Topics

### Consumed

| Topic | Consumer Group | Action |
|---|---|---|
| `twende.rides.completed` | `twende-rating` | Marks ride as rateable; stores ride metadata for later rating inference |

When a `RideCompletedEvent` is received, the service stores the ride's `rideId`,
`riderId`, `driverId`, and `countryCode` in a local cache or lookup table so that
when a rating is submitted, the service can resolve who is being rated without
calling ride-service synchronously.

## 9. Service Logic

### Submit Rating Validation

```java
public RatingDto submitRating(UUID userId, String role, SubmitRatingRequest request) {
    // 1. Validate ride exists and is COMPLETED
    RideDetails ride = resolveRide(request.getRideId());
    if (ride == null || !ride.getStatus().equals("COMPLETED"))
        throw new BadRequestException("Can only rate completed rides");

    // 2. Validate caller is part of this ride
    if (role.equals("RIDER") && !ride.getRiderId().equals(userId))
        throw new UnauthorizedException("You are not the rider on this ride");
    if (role.equals("DRIVER") && !ride.getDriverId().equals(userId))
        throw new UnauthorizedException("You are not the driver on this ride");

    // 3. Check for duplicate rating
    if (ratingRepository.existsByRideIdAndRaterRole(request.getRideId(), role))
        throw new ConflictException("You have already rated this ride");

    // 4. Determine who is being rated
    UUID ratedUserId = role.equals("RIDER") ? ride.getDriverId() : ride.getRiderId();

    // 5. Save rating
    Rating rating = new Rating();
    rating.setRideId(request.getRideId());
    rating.setRatedUserId(ratedUserId);
    rating.setRaterUserId(userId);
    rating.setRaterRole(role);
    rating.setScore(request.getScore());
    rating.setComment(request.getComment());
    rating.setCountryCode(ride.getCountryCode());
    ratingRepository.save(rating);

    // 6. Recompute and cache aggregate
    updateRatingAggregate(ratedUserId);

    // 7. Publish Kafka event
    ratingEventPublisher.publishRatingSubmitted(rating);

    return toDto(rating);
}
```

### Rating Window

Ratings should only be allowed within a reasonable window after ride completion (e.g.,
48 hours). Check `ride.completedAt` against the current time during validation.

## 12. Key Implementation Rules

1. **One rating per ride per role** -- enforced at both the database level (unique
   constraint) and the application level (check before insert).

2. **Score range 1-5 only** -- validated in the DTO with `@Min(1) @Max(5)` and in the
   database with a CHECK constraint.

3. **Ratings only for COMPLETED rides** -- reject rating submissions for rides in any
   other status. Check the ride status via the cached ride data or ride-service call.

4. **Caller must be a participant** -- the rater must be either the rider or the driver
   on the ride. Validate against `X-User-Id` header.

5. **Internal endpoint has no auth wrapper** -- `/internal/**` endpoints are called
   service-to-service and bypass the API Gateway. They are not exposed externally.

6. **Aggregate recomputation is synchronous** -- on each rating submission, recompute
   the aggregate and update Redis before returning the response.

7. **All responses use `ApiResponse<T>` wrapper** -- consistent with the platform-wide
   response format. Exception: internal endpoints return plain DTOs.

8. **No Eureka, no Feign** -- use Spring `RestClient` for all inter-service HTTP calls.

9. **Kafka key format** -- use `{countryCode}:{rideId}` as the Kafka message key for
   partition locality.

10. **Comments are optional** -- a rating can be submitted with just a score.

- [ ] 3. Repository: `RatingRepository` -- `existsByRideIdAndRaterRole`, `findAverageScoreByRatedUserId`, `countByRatedUserId`, distribution query by score
- [ ] 4. `RideClient`: RestClient to ride-service `GET /internal/rides/{id}` for ride validation (riderId, driverId, status, completedAt)
- [ ] 5. `RatingService`: `submitRating(userId, role, request)` -- validate ride COMPLETED, caller is participant, no duplicate (unique check), determine ratedUserId from role, save rating, recompute aggregate, update Redis cache `rating:driver:{id}` / `rating:rider:{id}`, publish Kafka event; `getDriverScore(driverId)` -- read from Redis, compute from DB on cache miss
- [ ] 6. Kafka consumer: `RideCompletedConsumer` listening on `twende.rides.completed` (group: `twende-rating`) -- store ride metadata locally for later rating inference (avoid sync call to ride-service)
- [ ] 7. Kafka producer: `RatingEventPublisher` publishing `RatingSubmittedEvent` to `twende.ratings.submitted` (key: `{countryCode}:{rideId}`)
- [ ] 8. `RatingController` (POST `/api/v1/ratings`, GET `/api/v1/ratings/driver/{driverId}`, GET `/api/v1/ratings/me`) + `InternalRatingController` (GET `/internal/ratings/driver/{driverId}/score` -- plain DTO, no `ApiResponse` wrapper)
- [ ] 9. Flyway migration: `V1__create_ratings_schema.sql` (table with UNIQUE(ride_id, rater_role), CHECK(score BETWEEN 1 AND 5), indexes)
- [ ] 10. Unit tests + integration tests (Testcontainers for PostgreSQL, Redis, Kafka; WireMock for ride-service), verify >= 80% coverage with `./mvnw -pl rating-service verify`
